# Agent workflow log

Running log of AI agent usage on this project — what each agent/session did, what failed, what was fixed by hand. This becomes the About-page writeup and interview material.

| Date | Stage | Agent/lane | What happened |
|---|---|---|---|
| 2026-07-10 | 5 | Single agent (Cowork) | Phase 1 scaffold: Next.js app, QuerySpec schema, /api/ask pipeline with mock mode, dashboard UI, CLAUDE.md. npm registry unavailable in sandbox — deps installed locally by hand. |
| 2026-07-14 | 5 | Single agent (Cowork) | Migrated styling from plain CSS to Tailwind v4 + shadcn/ui (Card/Table/Badge/Button/Input); diagnosed Gemini free-tier key returning 0-quota, switched provider to Anthropic; added 14 skills to the demo dataset. |
| 2026-07-14 | 5 | Single agent (Cowork) | Built `scripts/ingest-adzuna.ts` to replace static placeholder data with real Adzuna postings, ran it once credentials arrived. Discovered the 30-vs-60-day trend proxy was systematically biased upward (every skill showed 70-210% "growth") because Adzuna's `max_days_old` counts currently-live listings, and older listings get delisted faster than they accumulate — not a rounding issue, an actively misleading number. |
| 2026-07-14 | 5 | Single agent (Cowork) | User decided against the static-snapshot/ingest-script model entirely — wanted every number genuinely sourced from the API, not a periodically-regenerated file. Replaced with `src/lib/adzuna.ts`: live, uncached (`cache: "no-store"`) Adzuna fetch on every page request, deduped per-request via React `cache()`. Deleted `src/data/snapshot.ts` and `scripts/ingest-adzuna.ts`. Reverted the skill list to the original 8 (dropped the 14 added earlier) to keep per-request latency/quota usage sane — 8 skills × 3 calls, batched at concurrency 4. Trend proxy narrowed to 7-day-vs-prior-7-day (same bias, smaller window) since there's no longer a file to accumulate real history in. |
| 2026-07-14 | 5 | Single agent (Cowork) | Predicted risk landed fast: fully uncached Adzuna fetch hit a real 429 within a few page reloads (dev-mode HMR + manual refreshes). Fixed with two changes to `adzuna.ts`: (1) `next: { revalidate: 60 }` instead of `cache: "no-store"` — genuinely live (refreshes ~every minute) but survives repeated reloads; (2) 429 retry-with-backoff (honors `Retry-After`) plus per-skill error isolation (`Promise.allSettled` — one failing skill is dropped, not a page crash). `data.ts`/`StatCards.tsx` updated to handle an all-skills-failed edge case gracefully instead of crashing on `topTrend.skill`. |
| 2026-07-14 | 5 | Single agent (Cowork) | Recurring hydration warning traced to two `toLocaleString()` calls with no explicit locale (server/client locale can disagree) — pinned to `"en-US"`. Residual warning after that fix had no remaining code-level cause (grepped the whole tree), so added `suppressHydrationWarning` to `<html>`/`<body>` in `layout.tsx` — standard scoped fix for browser-extension DOM injection, the one remaining known cause. |
| 2026-07-14 | 5 | Single agent (Cowork) | User flagged the Ask bar's AI-generated insights as generic/low-context — root cause: `/api/ask`'s first (and only) AI call never saw real data, only skill names, so `insight` was an ungrounded guess. Built `src/lib/query.ts` (`executeQuerySpec`) to run the validated `QuerySpec` against live Adzuna data for real result rows, and added a **second** AI call in `route.ts` that writes the insight from those real numbers (falls back to a deterministic template if that call fails). `AskBar.tsx` now renders the actual `QueryResult` (table/statCard; `barChart` renders as a table until Recharts is added) instead of a raw JSON dump — closes out the long-standing TODO. Also dropped `filters.months` and `groupBy: "month"`/`visualization: "lineChart"` from what the AI is prompted to request, since there's no time-series data source to back them. Added a 9th skill, JavaScript, along the way. |
| 2026-07-16 | 6 | Single agent (Claude Code) | Added conversational refinement to the Ask bar so follow-ups ("now only remote roles") build on the previous turn instead of starting over — while keeping the "never a chat transcript" rule intact (still structured `QuerySpec` → `QueryResult` per turn, just stacked). Conversation state lives client-side only (last 3 `{question, spec}` turns, no server session); `route.ts` validates that history with a new `AskHistorySchema` and feeds it into the spec-generation prompt as merge/refine guidance. The insight call now also recomputes the previous turn's result server-side via `executeQuerySpec` (never trusts client-sent numbers) and passes it as comparison context. `AskBar.tsx` now renders a list of exchanges instead of one result, with a "New conversation" reset. Verified via direct `/api/ask` calls: a remote-only follow-up correctly merged into the prior spec, an unrelated follow-up correctly ignored history, and malformed history correctly 400s. Not verified in-browser — no browser automation tool available this session; UI code should get a manual click-through before considering this fully done. |
| 2026-07-16 | 6 | Single agent (Claude Code) | Reworked the just-shipped conversational refinement into a different interaction: after each answer, 3 AI-generated follow-up questions render as clickable chips (replacing the static suggestion chips once a conversation has started), and new answers now render at the **top** of the list instead of appended at the bottom. User explicitly split intent by input method: typing in the box always starts a fresh, history-free question; only clicking a follow-up chip carries the last 3 turns forward as refinement context — resolves the ambiguity the previous free-text-follow-up design left to the model's judgment. To avoid a 3rd AI call per turn, folded the follow-up suggestions into the existing insight call (`writeInsight` → `writeInsightAndFollowUps`, now returns `{insight, followUps}` as validated JSON via a new `InsightResponseSchema`) rather than adding a separate call — same 2-calls-per-turn cost as before. Added `fallbackFollowUps()`/`MOCK_FOLLOWUPS` alongside the existing `fallbackInsight()`/`MOCK_INSIGHT` for AI-failure and no-API-key paths. Mid-verification hit one live fallback (an Anthropic call returned generic template follow-ups instead of grounded ones) — traced it to a transient issue, not a code bug: a direct repro of the same prompt against the Anthropic API succeeded cleanly (170 output tokens, well under the 500 cap), and re-running the exact same live flow twice more succeeded both times with well-grounded, non-generic follow-ups. Confirmed the fallback path itself also works correctly (returned exactly what `fallbackFollowUps()` should produce for that spec) — so the failure mode is handled gracefully either way. Verified via direct `/api/ask` calls only; browser click-through of the new chip/ordering UI still owed (no browser automation tool available this session). |
